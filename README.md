# DocAgent-mini: AI‑агент для работы с документацией

**DocAgent-mini** — это учебный мини‑проект AI‑агента, который отвечает на вопросы пользователей по внутренней документации компании через HTTP‑API. Проект объединяет ключевые концепции современной разработки AI‑решений: RAG (Retrieval‑Augmented Generation), управление состоянием (stateful agents), безопасность и структурированное логирование.

## Цель проекта

Создать минимально жизнеспособный AI‑агент, который:
* ищет информацию в локальной документации (Markdown‑файлы);
* создаёт и обновляет задачи (тикеты);
* запоминает состояние для каждого пользователя между запросами;
* предоставляет простой HTTP‑интерфейс для взаимодействия;
* соответствует базовым стандартам продакшен‑кода (логирование, обработка ошибок, валидация).

Проект предназначен для закрепления навыков работы с LLM, RAG и stateful‑агентами, а также понимания приципов работы технологий langchain и langgraph. Его можно использовать как стартовую точку для создания корпоративного AI‑помощника.

## Технологический стек

* **Python 3.10+** — основной язык разработки;
* **FastAPI** — фреймворк для создания HTTP‑API;
* **Uvicorn** — ASGI‑сервер для запуска приложения;
* **Sentence Transformers** — генерация эмбеддингов текста;
* **Pydantic** — валидация данных и работа с моделями;
* **Loguru** — структурированное логирование;
* **pytest** — тестирование;
* **Flashrank** — ранжирование результатов поиска;
* **ollama** — запуск локальных LLM;
* **llama3.1** — локальная LLM;
* **langchain-core** — базовые абстракции и компоненты LangChain для построения AI‑агентов;
* **langchain-ollama** — интеграция Ollama с экосистемой LangChain, позволяющая использовать локальные модели в цепочках обработки;
* **langgraph** — библиотека для построения stateful‑агентов и сложных рабочих процессов (workflows) с поддержкой управления состоянием и ветвлением логики.

## Функциональные возможности

Агент обрабатывает три типа запросов:

1. **Информационные запросы (RAG)**
   * Пользователь: *«Что такое RAG?»*
   * Агент: ищет релевантные фрагменты в документации (`docs/*.md`) и формирует ответ.

2. **Создание задачи**
   * Пользователь: *«Создай задачу: добавить пример агента»*
   * Агент: вызывает `create_task`, сохраняет `task_id` в состоянии пользователя.

3. **Уточнение по задаче**
   * Пользователь: *«А теперь добавь комментарий: сделано»*
   * Агент: использует сохранённый `task_id` и вызывает `add_comment`.

## Установка

1. Клонируйте репозиторий:
   ```bash
   git clone <https://github.com/YuriiGaponov/DocAgent-mini.git>
   cd DocAgent-mini
```
2.  Создайте виртуальное окружение:
        ```
    python -m venv venv
    
    ```
3.  Активируйте виртуальное окружение:
    
    -   **Windows:**
        
        ```
        venv\Scripts\activate
        
        ```
        
    -   **macOS/Linux:**
        
        ```
        source venv/bin/activate
        
        ```
4.  Установите зависимости:
    
    ```
    pip install -r requirements.txt
    
    ```
5.  Запустите Ollama (если используете локальную LLM):
    
    ```
    ollama serve
    
    ```
    
6.  Подготовьте документацию: поместите Markdown‑файлы в папку `docs/`.
```
    

## Запуск

Запустите сервер с помощью Uvicorn:
uvicorn main:app --reload
Сервер будет доступен по адресу: `http://localhost:8000`

Запустите сервер Ollama:
ollama serve
Загрузите локальную модель (например llama3.1), если ещё не загружена
ollama pull llama3.1
Для проверки доступных моделей выполните:
ollama list


## Использование

### Доступные эндпоинты

-   `GET /health` — проверка работоспособности сервиса.
    
-   `POST /ask` — отправка вопроса пользователю.
    

### Пример запроса к `/ask`

**HTTP‑запрос:**

```
POST /ask HTTP/1.1
Host: localhost:8000
Content-Type: application/json

{
  "user_id": "user123",
  "query": "Что такое RAG?"
}

```

**Ответ:**

```
{
  "status": "success",
  "response": "RAG (Retrieval-Augmented Generation) — это подход, при котором модель сначала ищет релевантную информацию...",
}

```

### Пример с созданием задачи

**Запрос:**

```
POST /ask HTTP/1.1
Host: localhost:8000
Content-Type: application/json

{
  "user_id": "user123",
  "query": "Создай задачу: построить Звезду Смерти"
}

```

**Ответ:**

```
{
  "status": "success",
  "response": "Создана задача «Построить Звезду Смерти» с ID: 1."
}

```

### Пример с добавлением комментария

**Запрос:**

```
POST /ask HTTP/1.1
Host: localhost:8000
Content-Type: application/json

{
  "user_id": "user123",
  "query": "Добавь комментарий: Звезда Смерти построена"
}

```

**Ответ:**

```
{
  "status": "success",
  "response": "Добавлен комментарий «Звезда Смерти построена» к задаче ID 1."
}

```

```

```
pytest -v

Тесты покрывают:

-   обработку ошибок;
    
-   валидацию входных данных (Pydantic);
    
-   логирование (Loguru).
    

## Настройка

Основные настройки проекта находятся в `src/settings.py`. Вы можете изменить:

-   путь к папке с документацией (`DOCS_PATH`);
    
-   параметры векторной базы данных;
    
-   настройки LLM (модель, API‑ключ и т. д.);
    
-   параметры логирования (Loguru);
    
-   настройки ранжирования (Flashrank).
    

## Ограничения и допущения

1. **In‑memory хранилище состояний.** В целях экономии времени разработки для хранения состояний пользователей используется оперативная память (in‑memory). Это влечёт следующие ограничения:
   * все данные будут потеряны при перезапуске сервера;
   * решение не масштабируется на несколько инстансов приложения;
   * отсутствует резервирование и долгосрочное хранение данных.

2. **Локальная модель LLM.** Используется локальная модель `llama3.1` через Ollama, что накладывает требования к аппаратным ресурсам:
   * требуется не менее 16 ГБ ОЗУ для стабильной работы модели;
   * производительность зависит от вычислительных мощностей локальной машины;
   * возможны задержки в ответах при высокой нагрузке.

3. **Ограниченный формат входных данных.** Документация должна быть представлена в формате Markdown (`.md`) и размещена в папке `docs/`. Другие форматы файлов не поддерживаются.

4. **Упрощённая система задач.** Функционал создания и обновления задач (тикеты) реализован в минимальном объёме:
   * нет поддержки приоритетов задач;
   * отсутствуют статусы задач (например, «в работе», «завершена»);
   * не предусмотрена категоризация или тегирование задач.

5. **Базовое ранжирование результатов.** Для ранжирования релевантных фрагментов из документации используется библиотека Flashrank. Алгоритм обеспечивает базовое качество отбора, но не учитывает:
   * контекстную релевантность в сложных запросах;
   * персонализацию результатов под конкретного пользователя.

6. **Отсутствие аутентификации.** В текущей версии API не требует аутентификации — любой пользователь может отправлять запросы к эндпоинтам. Это делает проект непригодным для продакшена без доработки.

7. **Упрощённая обработка ошибок.** Система обрабатывает основные типы ошибок, но не покрывает все возможные сценарии сбоев (например, сетевые ошибки при взаимодействии с Ollama или проблемы с чтением файлов).

8. **Учебная направленность.** Проект разработан как учебный прототип, поэтому:
   * код оптимизирован для наглядности и обучения, а не для максимальной производительности;
   * некоторые компоненты (например, RAG‑пайплайн) могут требовать доработки для работы с большими объёмами данных;
   * архитектура допускает упрощения, которые в продакшене потребовали бы более сложных решений.

9. **Зависимость от локальных сервисов.** Работа агента требует запущенного сервера Ollama. Отсутствие подключения к Ollama приводит к неработоспособности AI‑функционала.

10. **Ограниченная валидация запросов.** Валидация входных данных реализована через Pydantic, но фокусируется на структуре запроса (`user_id`, `query`), не проверяя семантику содержимого (например, длину текста или запрещённые слова).
